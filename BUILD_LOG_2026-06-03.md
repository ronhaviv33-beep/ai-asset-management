# AIFinOps Guard — Build Log

## What We Built (03/06/26 → 05/06/26)

---

### The Product
**AIFinOps Guard** is an AI Runtime Intelligence Platform — every LLM call in your organization flows through it. It scans for PII, enforces budgets, enforces model policies, logs everything, runs per-team Guard Modes (Visibility First → Governance Later), and issues revocable API keys so customers can connect their own agents in two lines of code.

Work spanned three sessions: production wiring, client test environment, alert tuning, then a full multi-tenancy (BYOK) redesign.

---

## Phase 1 — Production Frontend → Backend Wiring

**Problem:** Login failed in production — `ERR_NAME_NOT_RESOLVED`. Dashboard couldn't reach the backend.

**Root causes (stacked):**
1. `render.yaml` injected `VITE_API_URL`; frontend read `VITE_API_BASE` — value never reached the bundle.
2. Render's `fromService` `property: host` resolved to the bare service name (`aifinops-backend`) not the full host (`aifinops-backend.onrender.com`).

**Fix:**
- `dashboard/src/api.js`: `resolveBase()` accepts either var name, strips trailing slashes, prepends `https://` to a bare host, falls back to `/api` in dev.
- `render.yaml`: switched `VITE_API_BASE` to `sync: false` — set manually in Render dashboard to the full `https://` URL.

## Phase 2 — Client Test Environment

**What was built:**
- `test_agent.py` — traffic-simulator: 5 realistic task types, normal call every 3 min, PII-laden call every 20 min, 8-hour runtime cap.
- Confirms end-to-end: `gk-` key → gateway → LLM → telemetry.

**Key clarification:** the simulator is not what customers run. A real customer changes two lines:

## Phase 3 — Telemetry Attribution Fix (`unknown / unknown`)

**Problem:** Calls with a `gk-` key and no `X-Guard-*` headers logged as `unknown / unknown`.

**Fix:** `_caller_team()` / `_caller_name()` helpers; both proxy routes now resolve:
```
team = X-Guard-Team header  OR  the API key's own team  OR  "unknown"
```
The key IS the identity — headers are optional labeling for multi-agent keys.

---

## Phase 4 — Alert Threshold Fix

**Problem:** Alerts page was empty despite real runaway-agent traffic because thresholds were unreachable.

**Fix:** Loop threshold 40 → 5 calls/30-min window; after-hours threshold 15 → 5; cost spike now fires on either 2× 7-day average OR $0.05 absolute, whichever is lower.

## Phase 5 — BYOK Multi-Tenancy (the architectural redesign)

### Why it was needed

The original model held one set of provider keys (`OPENAI_API_KEY` etc.) in platform env vars and shared them across all customers. Every customer request hit the **platform's** OpenAI/Anthropic/Google bill. That's the exact failure mode of every naive LLM proxy.

### The design

Each customer org brings their own provider key. The platform stores it encrypted. Every proxy call resolves the key for the caller's org and uses it. There is no fallback to platform keys — missing key = hard 4xx.

### Core security invariant

> **A non-platform org can never resolve to env-var keys, under any condition.**

Every design decision below was made to hold this invariant structurally, not just conditionally.

---

### New database schema

The platform now has **10 tables**:

| Table | Purpose |
|---|---|
| `users` | Dashboard users; `organization_id` FK (nullable only during migration) |
| `api_keys` | Issued `gk-` keys; `organization_id` FK (same) |
| `organizations` | **NEW** — tenant entity; `is_internal=True` for the platform org |
| `provider_credentials` | **NEW** — per-org encrypted provider keys; `UNIQUE(organization_id, provider)` |
| `telemetry` | Per-call log: team, agent, model, tokens, cost, PII findings |
| `budget_rules` | Team/agent spend limits |
| `guard_modes` | Per-team mode overrides (observe/alert/enforce) |
| `sessions` | Multi-turn chat sessions |
| `session_messages` | Messages within a session |
| `policy_rules` | Model allowlist/blocklist rules |

**Key structural choices:**
- `organization_id` on `users` and `api_keys` is nullable only during the backfill migration window. After migration it is non-null for all rows.
- `organization_id` is **never** embedded in the JWT. The token carries `user_id` only; org is resolved from the DB on every request — avoids the stale-token window.
- `provider_credentials.encrypted_key` is Fernet ciphertext. Plaintext never stored, never returned by any API.

---

### Encryption

Provider keys are encrypted at rest with Fernet symmetric encryption (`cryptography` library).

```python
# app/models.py
def _fernet() -> Fernet:
    raw = os.getenv("CREDENTIAL_ENCRYPTION_KEY", "")
    if not raw:
        raise RuntimeError("CREDENTIAL_ENCRYPTION_KEY is not set")
    return Fernet(raw.encode())

def encrypt_credential(plaintext: str) -> str:
    return _fernet().encrypt(plaintext.encode()).decode()

def decrypt_credential(ciphertext: str) -> str:
    return _fernet().decrypt(ciphertext.encode()).decode()
```

`CREDENTIAL_ENCRYPTION_KEY` is a Fernet key set in the Render dashboard (`sync: false`). Never committed.

---

### Boot migration (`app/migrate_orgs.py`)

Runs on every deploy. Fully idempotent.

1. Creates new tables if not present.
2. Seeds the platform org (`is_internal=True`, `slug="platform"`) if it doesn't exist.
3. Seeds the platform org's `provider_credentials` from env vars (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`), encrypted with Fernet — **only if no credential row exists yet for that org+provider pair**.
4. Backfills `api_keys` with `organization_id=NULL` → platform org.
5. Backfills `users` with `organization_id=NULL` → platform org.

Guards: each step checks existence before writing. Second boot touches nothing already set. A real customer key stored after first boot is never overwritten by the migration on subsequent boots.

---

### Auth enforcement

**`get_current_user`** (dashboard JWT path): hard 401 if `user.organization_id is None`.

**`get_proxy_caller`** (proxy path — JWT or `gk-` key):
- JWT: resolves `User` from DB, hard 401 if `organization_id is None`.
- `gk-` key: verifies SHA-256 hash against `api_keys` table, hard 401 if `organization_id is None`. Returns a dict including `"organization_id": api_key.organization_id`.

Neither path falls back to a default organization. Null org = hard 401, always.

---

### Org client resolution (`app/client.py`)

```python
def get_client_for_org(organization_id: int, model: str, db) -> AsyncOpenAI:
    provider = _provider_for(model)
    cache_key = (organization_id, provider)

    if cache_key in _org_clients:          # cache hit — (org_id, provider) tuple
        return _org_clients[cache_key]

    cred = db.query(ProviderCredential).filter(
        organization_id == organization_id,
        provider == provider,
        is_active == True,
    ).first()

    if cred is None:
        raise HTTPException(402, "No {provider} credential for org...")   # hard 402

    api_key = decrypt_credential(cred.encrypted_key)   # RuntimeError → 500 if decrypt fails

    # build provider-specific AsyncOpenAI client
    ...
    _org_clients[cache_key] = client
    return client
```

Cache is keyed by `(organization_id, provider)` — integer PK, not a string team name. No cross-org collision possible.

---

### Proxy route hardening

Both `/v1/chat/completions` and `/v1/messages`:

```python
org_id = _caller_org_id(current_user)
if org_id is None:
    raise HTTPException(401, "No organization resolved for this credential")
org_client = get_client_for_org(org_id, model, db)   # 402 if no credential
```

`proxy_chat_complete` and `proxy_chat_stream` **no longer accept `org_client=None`**. The `else _get_client(model)` fallback to env-var keys has been removed. If called without a resolved client, `proxy_chat_stream` raises `RuntimeError` immediately.

`_get_client(model)` has exactly two remaining callers: `complete()` and `chat_complete()`, used only by `/ask`, `/chat`, `/sessions/*/chat` — dashboard routes gated by JWT (`get_current_user`). Not reachable from any `gk-` key or customer proxy path.

### Provider credentials CRUD (`app/main.py`)

| Endpoint | Auth | Behavior |
|---|---|---|
| `GET /provider-credentials` | `require_admin` | Returns `last4`, `provider`, `is_active` — never ciphertext or plaintext |
| `POST /provider-credentials` | `require_admin` | Validates key with test call, encrypts, upserts. `organization_id` from `actor`, never from request body |

`invalidate_org_client(actor.organization_id, provider)` is called on both update and delete.

```bash
curl -i -X POST $URL/v1/chat/completions \

# PASS = 402 "No openai credential configured for organization..."
# FAIL = 200

# B — platform org with valid credential → 200
curl -i -X POST $URL/v1/chat/completions \
  -H "Authorization: Bearer gk-<platform-key>" \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o-mini","messages":[{"role":"user","content":"hi"}]}'

# C — org with its own key → 200, hits their provider account (verify in their dashboard)

# D — null org_id → 401
#   Force: sqlite3 /data/guard.db "UPDATE api_keys SET organization_id=NULL WHERE id=X"

# E — cross-tenant write → credential lands on actor's org, never on injected org_id
#   Use a key that passes validation so the write path is actually exercised.
#   After the call: SELECT organization_id FROM provider_credentials WHERE provider='openai'
#   and confirm zero rows for the injected org_id. A 422 from key validation only proves
#   the validator ran, not that tenancy isolation holds.
curl -i -X POST $URL/provider-credentials \
  -H "Authorization: Bearer <user-in-org-X>" \
  -H "Content-Type: application/json" \
  -d '{"provider":"openai","key":"<validating-key>","organization_id":<org-Y-id>}'
```

**A and E are the gates.** A returning 402 proves fail-closed. E writing to org X (not org Y) proves `organization_id` from the request body is ignored.

## Current Architecture

```
Customer Agent (OpenAI SDK)
    ↓  gk-key  /v1/chat/completions  (or /v1/messages)
FastAPI Backend  (Render, persistent SQLite at /data)
    ├── Auth: gk- hash → api_keys → organization_id (hard 401 if null)
    ├── Org client resolution: organization_id + provider → provider_credentials
    │       → decrypt → AsyncOpenAI(org's own key)  [hard 402 if missing]
    ├── PII Scanner
    ├── Policy Engine (team-scoped model allowlist/blocklist)
    ├── Budget Engine (daily/monthly limits)
    ├── Guard Modes (observe / alert / enforce, per-team, shadow blocks)
    ├── Circuit Breaker (5 failures/60s → trip → fail-open + critical log)
    └── LLM Router → org's OpenAI / Anthropic / Google / Local key
Dashboard (React + Vite)
```

### LLM call path — per-org key, no platform key in the customer path

```
gk-key ──► auth ──► org_id ──► provider_credentials (encrypted)
                                        │ decrypt
                                        ▼
                              AsyncOpenAI(org's sk-key)
                                        │
                                        ▼
                              OpenAI / Anthropic / Google
                              (billed to the org, not the platform)
```

## Watch-Items (must resolve before customer #2)

These are phase-appropriate stopgaps for single-operator dogfooding. Each one has an expiry date and a named replacement.

### W-1 — Tenancy hardening must fail closed, not just log

**Current state:** If `telemetry.organization_id` can't be hardened after backfill (null rows remain), `migrate_orgs.py` logs `ERROR: MIGRATION INCOMPLETE` and the server boots anyway. The state is also queryable via `GET /health → tenancy_hardened: false`.

**Why it's acceptable now:** Single operator, you'll read the log. The loud ERROR + health flag is observable enough for dogfooding.

**Why it's not acceptable for customer #2:** A passive log line and a health flag are not tenancy controls. An enterprise buyer asking "what prevents cross-org data exposure" does not want the answer to be "we log an error and check it."

**What replaces it (pick one before customer #2 is onboarded):**
- Option 1 — refuse to start: raise in `migrate_orgs.run()` if nulls remain after backfill. Render shows a failed deploy; the previous healthy version keeps serving. Customer traffic never reaches a build that knows its isolation is incomplete.
- Option 2 — gate the leaky surface: boot succeeds, but the four dashboard read paths return 503 ("tenancy verification incomplete") if `_TENANCY_HARDENED` is False. The proxy still works; only the cross-org-leaky surface is gated.

The `_TENANCY_HARDENED` flag in `main.py` is already the right hook for Option 2 — the four read endpoints check it and raise 503 if False. That's a one-day implementation once the decision is made.

### Authentication model (app/auth.py)

**JWT** — custom HS256 (no library), 8h expiry. `organization_id` never in token — always resolved fresh from DB on every request (prevents org-hopping if token is replayed after org reassignment).

**API keys** — `gk-{32 url-safe chars}`. Prefix stored for display; full key SHA-256'd for verification. Timing-safe compare. Shown once at creation.

**Hard 401 invariant** — both auth paths: if the resolved principal has `organization_id = NULL`, return 401. No fallback, no guest path.

**Rate limiting** — login: 5 attempts / 60s / IP. In-memory; evicted after 10k unique IPs.

---

### Multi-tenancy enforcement (BYOK)

```
Request arrives
  → auth resolves actor.organization_id (hard 401 if null)
  → get_client_for_org(org_id, model, db)
      → cache hit → return cached AsyncOpenAI
      → cache miss → query provider_credentials WHERE org_id = actor.org_id
          → no row → HTTP 402 "No {provider} credential configured"
          → row found → decrypt_credential(encrypted_key) → build AsyncOpenAI
          → cache under (org_id, provider)
  → proxy call using org's client — billed to org's key
```

**Credential rotation path:**  
`POST /provider-credentials` (upsert) → validates key → encrypts → writes → `invalidate_org_client(org_id, provider)` evicts dict entry → next request rebuilds from new ciphertext. No other cache layer; eviction is immediate.

**Tenancy gate:**

**W-4 detail:** The 5s `Promise.race` on `fetchMe` is mechanically the same fix as the roles-arm timeout, but the cost is asymmetric. A roles timeout degrades UI momentarily — worst case the user sees fewer pages until they refresh. A `fetchMe` timeout clears the stored token and bounces to the login screen, even if the session was perfectly valid and the server was just slow (Render free-tier cold starts routinely exceed 5s on first request after idle). "Server was slow" and "session is invalid" are conflated, and the result is an authenticated user seeing an unexplained logout.

The right fix before pilot: distinguish timeout from a real 401. On timeout, keep the token and show a transient error or retry — don't clear credentials. On a genuine 401 response, clear the token. The two cases currently collapse into the same `null`-return path in `fetchMe`. Consider a longer auth-arm timeout (10–15s to absorb cold starts) and a `timeout` flag on the return so the caller can decide between "retry" and "log out."

**This is live now.** If you or anyone hits a Render cold start, the app will log them out with no explanation. Know the cause before it reads as a bug.
